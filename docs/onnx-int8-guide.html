同样是 INT8，为什么 Netron 里两张图长得完全不一样？

PTQ 导出 .onnx 后，用 Netron 打开可能看到两种截然不同的计算图。本文说明 QDQ 与 QOperator 的区别、部署取舍，以及如何正确验证量化精度。

引子

第一次把 PTQ 导出的 .onnx 拖进 Netron，我一度以为软件坏了。

一张图里，QuantizeLinear 和 DequantizeLinear 像栅栏一样把 Conv、Relu 围在中间；另一张图却干净得多，直接是 ConvInteger、MatMulInteger，几乎看不到 DQ。文件大小差不多，名字都带 int8，精度测试也过了——但部署同事说：「这张能上 NPU，那张不行。」

后来才搞明白：INT8 不是只有一种写法。ONNX 里至少有两套表示，Netron 只是把它们画出来了。搞不清这个，后面选引擎、上板子、查精度，每一步都可能白忙。

先搞懂：量化到底在干什么

训练好的模型，权重和中间结果通常是 FP32——32 位浮点，小数精度高，占内存大，算得也慢。

部署到手机、摄像头、车机，往往要压成 INT8，8 位整数，大约只有 FP32 四分之一体积，很多芯片对整数还有专门电路。

但整数没有小数。不能把 3.14159 直接塞进 -128～127，得先定一套「档位和真实值怎么对应」的规则：

真实值 ≈ scale × (整数值 − zero_point)
参数含义
scale一格整数代表多少实际量
zero_point整数 0 落在真实坐标的哪里（不一定是数学上的 0）

PTQ（Post-Training Quantization，训练后量化）就是：模型已经训完，不再动结构和训练流程，只在导出阶段做 INT8 压缩。省事，工程里很常见；精度通常略逊于边训边量化的 QAT，但对很多场景够用。

下面说的两种图，都是 PTQ 之后可能出现的形态——数学上在做同一件事，ONNX 的表达方式却不同。

两个工具，各管一段

ONNX（Open Neural Network Exchange）是通用的模型交换格式，后缀 .onnx。PyTorch、TensorFlow 训出来的网络，可以转成 ONNX，再交给不同环境的推理工具去跑。

Netron 用来「看」这个文件。每个方块是一个算子（节点），箭头是数据流。常见算子：

算子作用
Conv卷积，CNN 里最核心的运算
Relu把负数截成 0，保留正数
MatMul矩阵乘法，全连接层、Transformer 里常见

量化之后，图里多出来的节点类型，才是本文重点。

第一种：QDQ——「先兑换，算完再换回来」

QDQ 指的是图里大量出现 QuantizeLinear（Q）和 DequantizeLinear（DQ）。

浮点 ──→ [Q] ──→ INT8 ──→ Conv / Relu ──→ INT8 ──→ [DQ] ──→ 浮点
              ↑ scale 在这                    ↑ scale 在这

浮点数据先过 Q，变成 INT8；中间仍是普通的 Conv、Relu（不是 Integer 版）；算完再经 DQ 变回浮点。Netron 里 Q 和 DQ 成对铺开，像在每个运算前后各加一道兑换窗口。

好处是参数外露。 scale、zero_point 挂在 Q、DQ 节点上，点开就能看、能改，和硬件侧对齐也方便。

很多 NPU、手机芯片的工具链，正是按「Q → 普通 Conv → DQ」这种套路做优化的：识别到这段模式，就把三步合成一个 INT8 卷积核，少搬内存、少调度，速度才上得去。

这一步叫 融合（Fusion）；合成后在芯片上跑的那段优化代码，一般叫 kernel（这里别和操作系统内核混淆，就是指某个算子在硬件上的实现）。

第二种：QOperator——「整条线直接收整数」

INT8 ──→ ConvInteger / MatMulInteger ──→ INT8
              ↑ scale 收在算子属性里

图面干净，没有成片的 DQ；scale、zero_point 塞进算子属性，Netron 里不如 QDQ 直观。

在 CPU 上用 ONNX Runtime（ORT）跑，两种格式往往都能工作。问题多半出在 NPU 和专用 Converter 上。

Converter 就是把 ONNX 再转成某家芯片能吃的格式，顺带做图优化和融合。常见工具：

工具典型场景
ORTCPU / 通用推理
TensorRT NVIDIA GPU
昇腾（Ascend）华为 NPU
SNPE高通骁龙平台

各家认的节点、融合规则不完全一样。很多工具链的融合规则是按 QDQ 写的——看到 ConvInteger，可能不认、融不进去。该快的 INT8 反而在量化节点之间来回拷贝，有时甚至比 FP32 还慢。

一张表，把差异捋清楚

QDQQOperator
Netron 长什么样满屏 Q / DQ，中间是普通 Conv满屏 ConvInteger 等整数算子
scale 在哪Q、DQ 节点上，一眼能看见算子属性里，不太直观
CPU + ORT通常没问题通常也没问题
NPU / 边缘芯片多数工具链认这个，融合空间大常不支持或融合失败

所以会出现一种挺反直觉的情况：同是 INT8，QOperator 在 CPU 上跑得顺，上 NPU 却翻车；QDQ 图看着乱，反而是上板子的稳妥选择。

验精度：别比 raw 字节

量化之后要验证：INT8 模型和 FP32 原版差多少。可以看输出向量的余弦相似度、最大误差，或任务指标（分类看 Top-1 是否明显掉点）。

新手容易踩的坑：拿 INT8 输出的字节直接和 FP32 浮点比。

INT8 存的是编码，不是「概率 0.95」那种物理含义上的数。必须先从最后一个 DequantizeLinear 出来，还原成浮点，再和 FP32 模型输出对比。这就是常说的 物理量纲——要比的是真实数值，不是 raw 字节。

❌ INT8 字节  ↔  FP32 浮点        （错误）
✅ INT8 → DQ → 浮点  ↔  FP32 浮点  （正确）

字节相近，scale 偏一点，反量化后误差可能很大；反过来，字节差不少，还原后反而可能接近。

实操上怎么选

服务器 CPU 或通用 GPU
两种格式都可以试，以 ORT / TensorRT 实测为准。

手机、嵌入式、NPU
导出前先翻 Converter 文档；多数更认 QDQ——尽量生成带 QuantizeLinear / DequantizeLinear、中间保留普通 Conv 的图，给工具链留出融合空间。

Netron 里认图

  • 满屏 Q/DQ + 普通 Conv → QDQ
  • 满屏 ConvInteger、MatMulInteger → QOperator

术语速查

术语一句话
FP3232 位浮点，训练常用，精度高
INT88 位整数，体积小、算得快
PTQ模型训完后再量化，不用重训
scale / zero_point整数与小数的换算参数
QDQ用 Q/DQ 节点包裹普通算子，参数可见
QOperator直接使用整数版算子，参数内嵌
Fusion多步合并成一步，减少内存与调度开销
Kernel某算子在特定硬件上的优化实现
DequantizeLinear把 INT8 还原为浮点，验精度必过此步

写在最后

INT8 不是「压缩完就完事」的一个标签，而是两种图、两套工具链习惯、一种正确的验精度方式。

再打开 Netron，不必纠结「为什么同一份模型有两种脸」——认图、对工具链、用 DQ 后的浮点验精度，三件事理顺，后面少踩很多坑。
